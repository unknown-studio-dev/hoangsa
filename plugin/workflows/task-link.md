## Task Link Detection

### URL patterns

```
linear.app/*/issue/*          вҶ’ Linear
*.atlassian.net/browse/*      вҶ’ Jira
app.clickup.com/t/*           вҶ’ ClickUp
github.com/*/issues/*         вҶ’ GitHub Issues
github.com/*/pull/*           вҶ’ GitHub PR
app.asana.com/0/*/            вҶ’ Asana
```

### Detection logic

1. Scan user's input text for URLs matching the patterns above
2. If found вҶ’ check `task_manager` config in `.hoangsa/config.json`
   - If not configured вҶ’ run `/serve` first-time setup (Step 2 of serve workflow)
   - If configured but provider doesn't match URL вҶ’ ask user if they want to add this provider
3. Fetch task details via MCP (same as `/serve` Pull Step 4b)

### Step 3b: Fetch and process attachments

After fetching task details, check if the task has attachments (from the `### Attachments` section in the fetched context).

**If attachments exist:**

1. Create attachments directory:
   ```bash
   mkdir -p "$SESSION_DIR/attachments"
   ```

2. Download each attachment:
   - **Validate URL:** Only download if scheme is `https://`. Never fetch `file://`, `http://localhost`, or RFC-1918 addresses (10.x, 172.16-31.x, 192.168.x, 169.254.x).
   - **Sanitize filename:** Strip path separators and traversal components before saving:
     ```bash
     SAFE_NAME=$(basename "<original_filename>" | tr -dc 'a-zA-Z0-9._-')
     # Reject if empty or starts with dot
     [ -z "$SAFE_NAME" ] || [[ "$SAFE_NAME" == .* ]] && SAFE_NAME="attachment_$(date +%s)"
     ```
   - Use `WebFetch` or `curl` to download the file from its validated URL
   - Save to `$SESSION_DIR/attachments/$SAFE_NAME`
   - Skip files larger than 50MB вҖ” note them in EXTERNAL-TASK.md but don't download
   - If download fails вҶ’ warn and continue with remaining attachments

3. Classify each downloaded file by extension:
   | Type | Extensions |
   |------|-----------|
   | image | .png, .jpg, .jpeg, .webp, .gif, .bmp, .svg |
   | video | .mp4, .mov, .webm, .avi, .mkv |
   | text | .txt, .log, .json, .csv, .md, .yaml, .yml |
   | other | everything else |

4. Process by type:

   **Images:** Note the local path вҖ” Claude reads images natively. Include path in EXTERNAL-TASK.md for reference.

   **Videos:** **Do NOT process here.** Just download and classify as `video`. The calling workflow's media detection step (e.g., fix Step 1b, menu Step 2e, audit Step 2e) will scan `$SESSION_DIR/attachments/` and trigger `visual-debug` analysis for all video files in one pass. This avoids duplicate processing.

   **Text files:** Read content and include in EXTERNAL-TASK.md under `### Text Attachments` as code blocks.

   **Other files:** Note filename and size in EXTERNAL-TASK.md вҖ” don't attempt to read.

5. Update session state to include attachments:
   ```json
   {
     "external_task": {
       ...existing fields...,
       "attachments": [
         {
           "filename": "<name>",
           "type": "<image|video|text|other>",
           "size_bytes": 12345,
           "local_path": "$SESSION_DIR/attachments/<name>",
           "url": "<original URL>"
         }
       ]
     }
   }
   ```

6. Add to EXTERNAL-TASK.md:
   ```markdown
   ### Attachments
   | File | Type | Size | Local Path |
   |------|------|------|------------|
   | screenshot.png | image | 245KB | $SESSION_DIR/attachments/screenshot.png |
   | demo.mp4 | video | 12MB | $SESSION_DIR/attachments/demo.mp4 |

   ### Media Analysis
   (Populated by the calling workflow's media detection step after visual-debug runs)
   - Montage: `pending вҖ” will be generated by media detection step`
   - Diff: `pending вҖ” will be generated by media detection step`

   ### Text Attachments
   #### error.log
   ```
   <file content>
   ```
   ```

**If no attachments:** Skip this step entirely вҖ” no directory creation needed.

**If MCP doesn't support attachments for this provider:** Skip silently вҖ” the task details without attachments are still valuable.

6. Store task reference in session state:

```json
{
  "external_task": {
    "provider": "<provider>",
    "task_id": "<id>",
    "task_url": "<url>",
    "title": "<title>",
    "description": "<description>",
    "original_status": "<status>",
    "labels": [],
    "linked_at": "<ISO 8601>"
  }
}
```

7. Save full context to `$SESSION_DIR/EXTERNAL-TASK.md`

### Show confirmation

```
рҹ“Ӣ Task linked: <provider> <task_id>
   <title>
   Status: <status>  Priority: <priority>
```

### What gets synced

At minimum (no user prompt needed):
- **Status change** вҶ’ "In Progress" when work starts, "In Review" or "Done" when work completes

User is asked (via `/serve` push Step 5c) for:
- **Comment** with work summary
- **Full report** with files/tests/commits

### Auto-set "In Progress"

When a linked task exists and a code-modifying workflow starts:

```bash
# At workflow start, if external_task exists:
# вҶ’ Call MCP to set status "In Progress" (non-blocking, best-effort)
```

This is silent вҖ” no user confirmation needed for "In Progress" since the user explicitly started working on it.

## Integration instructions for each workflow

### For `/menu`
- Check in Step 2d (before gathering requirements)
- Pre-fill task type from labels, description from task body
- Acceptance criteria from task carry to DESIGN-SPEC

### For `/fix`
- Check in Step 1 (gather bug context)
- Task description/comments become bug context
- Labels help identify affected layer (FE/BE/API)

### For `/cook`
- Check in Step 1 (load plan) вҖ” may already be linked from `/menu`
- Set "In Progress" at start

### For `/audit` and `/research`
- Check at start вҖ” use task context to scope the analysis
- No sync-back needed
